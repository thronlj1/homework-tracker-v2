# 班级作业轻量级扫码管家 - 核心需求文档 (完整 MVP 版)

### 一、 需求背景
**业务痛点：**
教师每日需高频统计班级多科目的作业上交情况。传统流程高度依赖课代表人工收集与教师手工勾选核对，耗时长、易漏记、进度不透明。同时，学校收作业的日程受法定节假日、调休、寒暑假及突发事件影响，存在复杂的时效要求。

**业务目标：**
利用教室现有的希沃白板设备及低成本扫码枪，结合粘贴于作业本的固定实体二维码，实现作业提交的“自助化”与“防呆化”。统计结果实时同步至教师手机端，支持可视化监控、快速催收、历史预警，并能自动适应复杂的教学日历。

**技术限制与选型依据：**
系统采用“零自持服务器”架构（Serverless）。前端分为采集端与管理端，通过第三方云数据库（Supabase）交互，不依赖传统的后端独立服务，以极低成本满足常态化运行。

---

### 二、 系统架构概述
本系统由三个核心模块构成：
1. **学生端（采集器）：** 运行于教室希沃白板（Windows 系统）的 Electron 桌面应用。
2. **教师端（管理台）：** 运行于教师个人手机的安卓应用（通过前端套壳技术打包）。
3. **数据中心（云端）：** 采用 Supabase 提供 PostgreSQL 数据库存储与 API 服务。

---

### 三、 学生端核心需求（希沃白板/采集端）
**定位：** 无人值守、极简交互、防误触、自动鉴权。

* **时间口径：** 以下所有“今日/工作日/时段”判断均以**北京时间（Asia/Shanghai）**为准。

1. **静默扫码接收功能**
   * **全局监听：** 软件开启全屏模式，全局监听扫码枪模拟的键盘输入事件及回车键。
   * **指令解析：** 识别约定的二维码字符串格式，提取班级、科目与学生标识。约定口径为：**每个“学生-科目”一张二维码**（同一学生有几门课，就有几张二维码）。
     - 二维码编码示例：`Class_<class_id>_Subject_<subject_id>_Student_<student_id>`
     - 数据隔离：采集端按二维码中的 `class_id` 写入数据库相关字段，保证跨班级数据不会串联。

2. **状态判定与视听反馈功能**
   * **成功录入：** 写入数据库后，界面弹出大字提示（如“✅ 张三 - 语文提交成功”），播放成功提示音。
   * **重复拦截：** 若该生当日已交该科，拒绝录入，显示红色警告（“❌ 请勿重复提交”），播放警告音；**不写入失败/重复记录**（仅前端提示与交互防呆）。
   * **异常拦截：** 扫入非约定格式条码时，提示“无效二维码”。

3. **多级时间守卫功能（核心自动化校验）**
   采集端在每日唤醒或发生扫码动作时，严格按以下优先级校验当前是否允许收作业：
   * **优先级 1（全局状态）：** 校验数据库中的学期状态。若为“寒暑假/长假暂停”，界面永久锁定，显示“当前为休假状态，系统已关闭”。
   * **优先级 2（当日人工干预）：** 校验教师是否对今日下达了强制指令。若为“强制关闭”，界面锁定显示“今日免交”；该日期视为**免交日（正向状态）**，后续扩展中不计入“负记录/惩罚/预警”。若为“强制开启”，直接放行。
   * **优先级 3（法定日历 API 校验）：** 在未受人工干预的情况下，自动调用第三方节假日 API（中国法定节假日规则）。若返回“周末”或“法定节假日”，界面锁定显示“今日为法定休息日，无需提交”；若教师对当天设置了覆盖安排，则**以教师安排为准**。
   * **优先级 4（每日时段校验）：** 若上述校验均通过（确认为工作日），系统进一步校验当前时间是否在“有效收作业时间段”（默认 `07:00 - 12:00`）内。超时则锁定界面显示“今日收作业已结束”。手工设置时，收作业起止时间**不允许跨天**（即 start 必须小于 end，且在同一天内）。
   * **锁定态行为（防误触）：** 当系统处于任意“锁定态”（如休假/今日免交/周末或法定休息/收作业已结束），页面仅展示对应原因；**不需要解析二维码字符串，也不进行写入**。

### 四、 教师端核心需求（手机 App/管理端）
**定位：** 实时监控、异常处理、效率工具、规则配置。

1. **今日数据看板**
   * **全局进度：** 首屏直观展示当日各科目的整体提交进度。
   * **明细列表：** 点击单科进入详情，分设“已交名单”与“未交名单”视图。

2. **高频快捷操作**
   * **一键催交：** 自动读取未交名单，生成催交文案写入手机剪贴板。
   * **特殊状态标记（免交/豁免）：** 针对请假或免交学生，教师可将其标记为“豁免/免交”。免交是一个**可记录的正向状态**：维度为**学生 + 科目 + 日期**，并用于后续扩展中避免将该天计入“负记录/惩罚/预警”。（即免交不会被当作应交未交来累计预警。）
   * **异常撤销：** 支持单条提交记录的撤销，处理代扫错扫事件；撤销后若仍处于“整体开放且在有效收作业时间段内”，允许学生再次提交（不需要锁死）。

3. **学情预警**
   * 系统自动检索数据，对连续 $N$ 天（可配置）的“应交未交”某科作业情况的学生打上红色预警标签。连续性的口径为：**只统计应交未交的连续工作日**；周末/中国法定节假日/今日强制关闭（免交日）/教师标记的豁免（免交）不计入连续未交天数。（例如周五与周一没交，连续计数为 2 天。）

4. **系统规则与日历控制台**
   * **全局任务开关：** 一键切换“学期中（进行）”与“寒暑假（暂停）”状态。
   * **今日状态覆盖：** 提供“跟随日历自动”、“今日强制开启（如补课）”、“今日强制关闭（如运动会）”三个选项，仅当日有效。
   * **基础规则：** 修改每日有效收作业起止时间，维护基础的科目表与学生名单。

### 五、 云端数据中心需求（Supabase）
**定位：** 数据存储与多端状态同步。

1. **核心数据表设计**
   * `classes`: 班级表（基础信息 + 班级图）。
   * `students`: 学生名册表（基础信息 + 学生头像，需归属 `class_id`）。
   * `subjects`: 科目表（收缴科目定义 + 科目图，需归属 `class_id`）。
   * `homework_records`: 作业提交流水表（用于已提交记录：记录单次扫码时间、状态、归属日期，并归属 `class_id` + `student_id` + `subject_id`）。
   * `homework_exemptions`: 免交/豁免记录表（教师“免交”标记用：记录学生 + 科目 + 日期，并归属 `class_id`）。
   * 图片字段（用于前端扩展渲染）：
     - `classes`：`class_image`（班级图）
     - `students`：`avatar_image`（学生头像）
     - `subjects`：`subject_image`（科目图）
   * 多班级支持要求：上述核心表需具备 `class_id`（或等价班级标识），并保证数据隔离在班级维度内。
2. **系统配置表（`system_configs`）扩展**
   * 存储每日时段（`scan_start_time`, `scan_end_time`）、预警阈值（`alert_continuous_days`）。
   * 存储多级时间守卫的控制变量（`global_task_status`, `today_override_date`, `today_override_status`），确保学生端与教师端的逻辑统一；并需在班级维度支持（例如带 `class_id`），确保教师对某班级的覆盖安排只影响该班级。
